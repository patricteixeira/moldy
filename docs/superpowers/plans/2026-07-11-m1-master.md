## Contratos fixados

### Perfis de formato (dimensões canônicas, CSS px)

```
post-1x1    1080 × 1080   safe area 48
post-4x5    1080 × 1350   safe area 48
story-9x16  1080 × 1920   safe area 64
doc-a4       794 × 1123   safe area 76   (A4 @ 96dpi; PDF via print)
```

### Artefatos JSON (schemas publicados em `schemas/`, gerados pelo motor)

- **Brand IR** — `schemas/brand-ir.schema.json`. Raiz: `schemaVersion, brand, revision, colors, fonts, roles, assets, formatProfiles, diagnostics`. Chaves de token: `color.primary|secondary|background|text`, `font.heading|body`. Roles: `heading|body|caption`; assets também carregam evidência. JSON sempre camelCase (`by_alias`).
- **Layout Spec** — `schemas/layout-spec.schema.json`. `{id, profile, namePt, canvas:{widthPx,heightPx,safeAreaPx}, background, slots:[{id, kind:"text"|"image"|"logo", role?, maxChars?, minResolution?, area:[x,y,w,h] (px), fit, required}]}`.
- **Content Spec** — `schemas/content-spec.schema.json`. `{layoutId, brandRevisionId, values: {slotId: {kind:"text",text} | {kind:"image",path,sha256?}}}`.
- **Guard Verdict** — `schemas/guard-verdict.schema.json`. `{checks: [{id, slotId?, status:"pass"|"fixed"|"blocked", messagePt, detail}]}`. Status `fixed` = corrigido automaticamente; `blocked` = exige ação do usuário. Nunca truncar/alterar conteúdo silenciosamente.

### Página de render (contrato Plano 2 ↔ Plano 3/4)

- Rota estática `render.html` da biblioteca TS; recebe payload por `window.__PAYLOAD__ = {brandIr, layoutSpec, contentSpec, assetsBaseUrl}` injetado antes do script.
- Renderiza o canvas do perfil em px 1:1 num elemento `#canvas` com `width/height` exatos.
- Sinaliza `window.__RENDER_DONE__ = true` somente após `document.fonts.ready` e layout estável.
- Assets de imagem/logo resolvem em `${assetsBaseUrl}/${path}`; fontes locais em
  `${assetsBaseUrl}/fonts/${fileSha256}`. Somente o placeholder constante
  `data:image/png;base64,...` pode ser usado verbatim; URLs `http:`, `https:`,
  protocol-relative, `blob:` e `javascript:` são recusadas pelo render.
- Publica medições internas em `window.__GUARD_REPORT__ = {overflows: [{slotId, contentPx, boxPx}], fontFallbacks: [{slotId, token, family, reason}]}`. `reason` distingue substituição confirmada de falha ao carregar arquivo.
- Payload ausente/inválido define `window.__RENDER_ERROR__` e nunca sinaliza `__RENDER_DONE__`.
- `renderDocumentStable` é o pipeline único do preview e da página: fontes usadas → segundo fitting → decode de imagens → dois frames. Há prova pixel a pixel, tolerância zero, no Chromium pinado.
- Export (Python + Playwright): viewport = dimensões do perfil, aguarda `__RENDER_DONE__`, converte medições + checks estáticos em `GuardVerdict` e bloqueia antes de publicar bytes; screenshot → PNG; `doc-a4` usa `page.pdf(format:"A4")`.

### Motor (contrato Plano 1 ↔ Plano 3)

Pacote Python `brand_runtime` em `packages/engine`, API pública:

```python
build_draft(package_dir: Path) -> BrandDraft            # extração + perguntas do wizard
compile_ir(draft, answers, brand_name) -> BrandIR       # confirmações → revisão imutável
generate_kit(ir: BrandIR) -> list[LayoutSpec]
run_static_checks(ir, layout, content, assets_dir) -> list[GuardCheck]
```

CLI de desenvolvimento: `brandrt extract|compile|kit|guard|schemas` (artefatos
JSON arquivo→arquivo; `guard` imprime um `GuardVerdict`). O Plano 2 acrescenta
`brandrt export` sem alterar os comandos existentes.

### API (contrato Plano 3 ↔ Plano 4) — v0 reconciliado

```
POST /v1/brands/imports        multipart pacote → {draftId, questions[], diagnostics, ignoredEntries}
POST /v1/drafts/{id}/compile   {answers, brandName} → {brandRevisionId}
GET  /v1/drafts/{id}/assets/{path} → asset sanitizado do draft
GET  /v1/brand-revisions/{id}  → Brand IR
GET  /v1/brand-revisions/{id}/kit → LayoutSpec[]
GET  /v1/brand-revisions/{id}/assets/{path} → asset da revisão
POST /v1/assets                multipart imagem → {sha256, size}
GET  /v1/assets/{sha256}       → imagem de conteúdo ou export
POST /v1/documents             {layoutId, brandRevisionId, values} → {documentId, checks}
POST /v1/documents/{id}/exports {format:"png"|"pdf"} → {jobId}
                                ou HTTP 409 {detail, checks} se houver blocked
GET  /v1/jobs/{id}             → {id, status, result, checks, error}
GET  /healthz                  → {status:"ok"}
GET  /v1/ping                  → {pong:true}
```

Tudo sob `/v1` exige `Authorization: Bearer <token>`; token por query string é
recusado. A UI usa somente paths same-origin e o proxy do Vite/nginx injeta o
header também para imagens e downloads. Storage content-addressed em
`sha256/ab/cd/<hash>`. Jobs: tabela em Postgres com status
`queued|running|succeeded|failed`; `checks` começa com o Guard estático e recebe
as medições do render. Um bloqueio medido deixa o job `failed`, sem publicar
arquivo, e a UI deve reapresentar esses checks no Guard Panel.

## Regras de execução

- Branch: `m1-walking-skeleton`. Commit por tarefa. Suíte completa + `ruff check` antes de cada commit.
- Windows é o ambiente de dev: paths via `pathlib`, nada POSIX-only. Sem `uv`/`pnpm`: usar `venv`+`pip` e `npm`.
- Strings visíveis ao usuário: PT-BR. Identidade e artefatos funcionais são
determinísticos; timestamps de auditoria são permitidos, injetáveis e
neutralizados no cálculo do id de revisão conforme a ADR 0006.
- Disciplina herdada do roadmap anterior: nada de expansão antes do skeleton fechar de ponta a ponta.
