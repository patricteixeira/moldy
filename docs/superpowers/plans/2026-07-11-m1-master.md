## Contratos fixados

### Perfis de formato (dimensões canônicas, CSS px)

```
post-1x1    1080 × 1080   safe area 48
post-4x5    1080 × 1350   safe area 48
story-9x16  1080 × 1920   safe area 64
doc-a4       794 × 1123   safe area 76   (A4 @ 96dpi; PDF via print)
```

### Artefatos JSON (schemas publicados em `schemas/`, gerados pelo motor)

- **Brand IR** — `schemas/brand-ir.schema.json`. Raiz: `schemaVersion, brand, revision, colors, fonts, roles, assets, formatProfiles, diagnostics`. Chaves de token: `color.primary|secondary|background|text`, `font.heading|body`. Roles: `heading|body|caption`. JSON sempre camelCase (`by_alias`).
- **Layout Spec** — `schemas/layout-spec.schema.json`. `{id, profile, namePt, canvas:{widthPx,heightPx,safeAreaPx}, background, slots:[{id, kind:"text"|"image"|"logo", role?, maxChars?, minResolution?, area:[x,y,w,h] (px), fit, required}]}`.
- **Content Spec** — `{layoutId, brandRevisionId, values: {slotId: {kind:"text",text} | {kind:"image",path,sha256?}}}`.
- **Guard verdict** — `{checks: [{id, slotId?, status:"pass"|"fixed"|"blocked", messagePt, detail}]}`. Status `fixed` = corrigido automaticamente; `blocked` = exige ação do usuário. Nunca truncar/alterar conteúdo silenciosamente.

### Página de render (contrato Plano 2 ↔ Plano 3/4)

- Rota estática `render.html` da biblioteca TS; recebe payload por `window.__PAYLOAD__ = {brandIr, layoutSpec, contentSpec, assetsBaseUrl}` injetado antes do script.
- Renderiza o canvas do perfil em px 1:1 num elemento `#canvas` com `width/height` exatos.
- Sinaliza `window.__RENDER_DONE__ = true` somente após `document.fonts.ready` e layout estável.
- Publica medições em `window.__GUARD_REPORT__ = {overflows: [{slotId, contentPx, boxPx}]}` para checks dependentes de medição de texto.
- Export (Python + Playwright): viewport = dimensões do perfil, aguarda `__RENDER_DONE__`, `screenshot` → PNG; `doc-a4` usa `page.pdf(format:"A4")`. O preview do editor e esta página usam **os mesmos componentes TS** — WYSIWYG por construção.

### Motor (contrato Plano 1 ↔ Plano 3)

Pacote Python `brand_runtime` em `packages/engine`, API pública:

```python
build_draft(package_dir: Path) -> BrandDraft            # extração + perguntas do wizard
compile_ir(draft, answers, brand_name) -> BrandIR       # confirmações → revisão imutável
generate_kit(ir: BrandIR) -> list[LayoutSpec]
run_static_checks(ir, layout, content, assets_dir) -> list[GuardCheck]
```

CLI de desenvolvimento: `brandrt extract|compile|kit|guard` (JSON in/out).

### API (contrato Plano 3 ↔ Plano 4) — esboço v0

```
POST /v1/brands/imports        multipart pacote → {draftId, questions[]}
POST /v1/drafts/{id}/compile   {answers, brandName} → {brandRevisionId}
GET  /v1/brand-revisions/{id}  → Brand IR
GET  /v1/brand-revisions/{id}/kit → LayoutSpec[]
POST /v1/documents             {layoutId, brandRevisionId, values} → {documentId}
POST /v1/documents/{id}/exports {format:"png"|"pdf"} → {jobId}; GET /v1/jobs/{id} → status+URL
```

Storage: content-addressed `sha256/ab/cd/<hash>` (herdado do system design anterior). Jobs: tabela em Postgres com status `queued|running|succeeded|failed` (decisão da spec §10: o mais simples suficiente).

## Regras de execução

- Branch: `m1-walking-skeleton`. Commit por tarefa. Suíte completa + `ruff check` antes de cada commit.
- Windows é o ambiente de dev: paths via `pathlib`, nada POSIX-only. Sem `uv`/`pnpm`: usar `venv`+`pip` e `npm`.
- Strings visíveis ao usuário: PT-BR. Determinismo: nada de relógio/aleatoriedade em outputs, exceto injetado.
- Disciplina herdada do roadmap anterior: nada de expansão antes do skeleton fechar de ponta a ponta.
