**Goal:** fechar o walking skeleton do M1 (spec §6): app web React+Vite em `apps/web` com o wizard de confirmação (uma pergunta por vez, mostra em vez de descrever) e o Slot Editor (kit com thumbnails, formulário por slots, preview ao vivo com a mesma biblioteca de render do export, guard com ação clara, exportar PNG/PDF); `docker-compose.yml` na raiz subindo tudo com `docker compose up`; e o teste E2E Playwright do roteiro completo da spec §6/M1 com um pacote de marca fixture.

**Architecture:** SPA React (React Router) que só fala com a API por caminhos relativos `/v1/...` (dev: proxy do Vite → `localhost:8000`; produção: nginx no container `web` faz proxy para `api`). O preview do editor e as thumbnails do kit montam **a mesma biblioteca de render** do export (`@brand-runtime/render`, Plano 2) — WYSIWYG por construção (spec §5.7, contrato do plano-mestre). Estado com `useReducer`/context — sem state manager externo. O guard estático roda no servidor (`POST /v1/documents` devolve `{documentId, checks}` — checks na forma do verdict do plano-mestre); o cliente só dá dicas locais (contador de caracteres) e **nunca** trunca ou altera conteúdo.

**Tech Stack:** TypeScript, React 19, Vite 7, React Router 7, jszip (monta o ZIP do pacote no navegador — contrato B), Vitest + Testing Library (jsdom), Playwright (`@playwright/test`) para E2E, nginx + Docker Compose. npm com Node 24 (sem pnpm). Fixture E2E gerada em Python pelo venv do engine (sem binários commitados).

**A. Biblioteca de render (Plano 2), consumo pelo editor:**

O Plano 2 fixa `renderDocument` (passe imediato), `renderDocumentStable` (pipeline autoritativo assíncrono) e `parsePayload`. Preview e export usam obrigatoriamente `renderDocumentStable`; o adapter local cancela uma estabilização antiga quando chega update novo:

```ts
// apps/web/src/render/mount.ts — adapter DESTE plano (não é export do Plano 2)
export interface RenderPayload {           // MESMA forma de window.__PAYLOAD__ (fixado no mestre)
  brandIr: unknown
  layoutSpec: unknown
  contentSpec: unknown
  assetsBaseUrl: string
}
export interface RenderHandle { update(payload: RenderPayload): void; destroy(): void }
export function mountRender(el: HTMLElement, payload: RenderPayload): RenderHandle
// mount/update → renderDocumentStable com AbortController; destroy aborta e esvazia
```

- Pacote npm `@brand-runtime/render` consumível via dependência `file:../../packages/render` (npm suporta `file:`; sem workspaces na raiz).
- Resolução de assets **fixada no Plano 2**: imagem/logo em `${assetsBaseUrl}/${path}` com `encodeURIComponent` por segmento — por isso `revisionAssetsBaseUrl` retorna a base **sem** barra final; fonte `source="file"` em `${assetsBaseUrl}/fonts/${fileSha256}`.
- O único path verbatim é o placeholder constante `data:image/png;base64,...`. URLs externas, protocol-relative, `blob:`, `javascript:` e traversal são recusados pelo Plano 2; assets reais sempre passam pelo storage controlado.

```ts
export interface RenderPayload {
  brandIr: unknown
  layoutSpec: unknown
  contentSpec: unknown
  assetsBaseUrl: string
}
export interface GuardReport {
  overflows: Array<{ slotId: string; contentPx: number; boxPx: number }>
  fontFallbacks: Array<{
    slotId: string; token: string; family: string;
    reason: "referenced-only" | "configured-fallback" | "load-failed"
  }>
}

export const mounts: Array<{ el: HTMLElement; payloads: RenderPayload[] }> = []

export function renderDocument(el: HTMLElement, payload: RenderPayload): GuardReport {
  const record = mounts.find((m) => m.el === el)
  if (record) {
    record.payloads.push(payload)
  } else {
    mounts.push({ el, payloads: [payload] })
  }
  el.setAttribute("data-render-stub", "1")
  return { overflows: [], fontFallbacks: [] }
}

export function parsePayload(raw: unknown): RenderPayload {
  return raw as RenderPayload
}

export async function renderDocumentStable(
  el: HTMLElement, payload: RenderPayload, _options?: { signal?: AbortSignal },
): Promise<GuardReport> {
  return renderDocument(el, payload)
}
```

- Regras normativas do reducer:
  1. `draft-created` em `upload` → `question` com `index: 0, answers: {}`; se `questions` vier vazio → direto `publish`;
  2. `answer` grava `answers[questionId] = value` e avança `index`; na última pergunta → `publish` (levando `answers`);
  3. `skip` só tem efeito quando a pergunta corrente tem `required === false`: avança sem gravar; em pergunta obrigatória devolve o **mesmo objeto** de estado (no-op, `===`);
  4. `back` em `question` com `index > 0` decrementa; com `index === 0` é no-op (`===`); em `publish` volta à última pergunta;
  5. `published` em `publish` → `done`;
  6. ações fora do passo esperado são no-op (`===`).

- **Cobertura do roteiro spec §6/M1:** passo 1–2 (instalar + confirmar + publicar revisão) → Tasks 3–5 e E2E; passo 3 (kit) → Task 6; passo 4 (post e documento por slots) → Task 7; passo 5 (guard intercepta violações semeadas com correção pelo usuário) → Task 8 + violações semeadas no E2E (texto 200 > 140 chars; foto 200×200 < 1080×1080); passo 6–7 (export PNG/PDF) → Tasks 8–9 e asserções de assinatura/dimensão no E2E; `docker compose up` → Task 9. "Export = preview" (igualdade de pixels) é critério do Plano 2 no CI — o E2E daqui verifica formato/dimensão, não pixels, de propósito (pixel-diff cross-plataforma no host seria flaky e duplicaria o teste do Plano 2).
- **Fidelidade às personas:** wizard mostra-não-descreve (cor como amostra sem hex visível — teste da Task 4; logo renderizado de verdade; fonte com amostra na própria família, carregando o arquivo real quando existe); editor sem jargão com teste automatizado (Task 7) cobrindo hex/px/token/nome de fonte; guard nunca trunca (testes das Tasks 7/8 verificam texto intacto) e toda mensagem vem do servidor em PT-BR com ação de um clique.
- **Contratos:** tudo que o plano-mestre fixa é consumido verbatim. O renderer fornece `renderDocument`/`parsePayload`, assets controlados e placeholder `data:image/png;base64`; a API fornece import ZIP, compile, assets de draft/revisão, documentos com checks, gate 409 e jobs `{id, status, result, checks, error}`. O painel atualiza com checks medidos do worker, inclusive quando um overflow falha o job sem publicar arquivo. Auth header-only via proxy same-origin foi ratificada para o M1.
- **Type-consistency interna:** `fakeApi.ts` (Task 2) fornece IR/layouts com os números do gerador do engine (Plano 1 Task 13) e é usado idêntico nas Tasks 3–8; testids da tabela normativa são os mesmos usados nos testes de componente e no E2E; os prompts PT-BR dos testes são os exatos do engine (Plano 1 Task 9 regra 9); as mensagens de guard dos testes são as exatas do engine (Plano 1 Task 14).
- **Riscos conhecidos e mitigação no próprio plano:** (a) ordem das perguntas/aparição de `color.secondary` no E2E não é assumida — o loop responde o que vier, com teto; (b) npm usa lockfiles e o worker fixa Playwright Python/imagem em 1.61.0; (c) a porta do `web` é a superfície confiável da instância single-tenant — trade-off consciente a revisitar quando houver multi-tenant; (d) guard medido é assíncrono, então um job aceito pode terminar `failed`, mas a UI recebe seus checks acionáveis.
- **Placeholders:** nenhum; único valor deliberadamente não-literal é o base64 de `PLACEHOLDER_IMAGE` (Task 6), especificado por procedimento determinístico (PNG 1×1 cinza gerado uma vez) e coberto por teste de forma (`data:image/png;base64,`).
