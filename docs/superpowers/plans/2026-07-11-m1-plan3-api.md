>
> **Formato deste plano:** os testes de cada tarefa são o contrato completo e obrigatório — implemente por TDD até que passem sem alterá-los (mudança em teste = desvio a reportar). Assinaturas, regras e tabelas dadas aqui são normativas. Onde o corpo da implementação não está escrito, ele é livre desde que os testes e as regras sejam satisfeitos.

**Goal:** a API HTTP do walking skeleton: embrulhar o motor do Plano 1 (`brand_runtime`) atrás dos endpoints do esboço v0 do plano-mestre — importar pacote ZIP → draft → compile → Brand IR → kit → documento com guard → export como job — com Postgres (SQLAlchemy), storage content-addressed e auth mínima por token de convite.

**Architecture:** app FastAPI em `apps/api` (pacote Python `brand_api` em `apps/api/src/brand_api`), modular monolith (spec §5.9). A API **não reimplementa nada do motor**: importa `brand_runtime` (instalado editable) e persiste/expõe seus artefatos. Postgres guarda metadados + JSONB de draft/IR/kit/content; blobs (uploads de imagem, exports) vão para storage content-addressed `sha256/ab/cd/<hash>`; o pacote de marca descompactado vive em diretório de trabalho persistente por draft (`<dataDir>/packages/<draftId>/`) para o motor operar arquivo→arquivo. Jobs de export são linhas na tabela `jobs` (decisão da spec §10: o mais simples suficiente), executadas por um worker que faz poll com `FOR UPDATE SKIP LOCKED`; o exporter real é o do Plano 2 (Playwright), atrás de uma interface injetável para os testes não dependerem de Chromium.

**Extensões aditivas ao esboço v0 do plano-mestre** (não alteram nenhum contrato existente; o orquestrador registra no plano-mestre ao fechar o M1 — este plano não edita docs):

- `GET /healthz` (sem auth) e `GET /v1/ping` (com auth — validação de token pelo Plano 4);
- `POST /v1/assets` + `GET /v1/assets/{sha256}` — upload/serviço de imagens de conteúdo e download de exports;
- `GET /v1/drafts/{id}/assets/{path}` — assets já sanitizados do draft para as perguntas visuais do wizard;
- `GET /v1/brand-revisions/{id}/assets/{path}` — assets do pacote da marca (logo) para preview;
- `POST /v1/documents` responde `{documentId, checks}` (o guard roda na criação; criação nunca é bloqueada — o bloqueio é do export);
- `POST /v1/brands/imports` responde também `diagnostics` e `ignoredEntries`;
- auth: header `Authorization: Bearer <token>` em tudo sob `/v1`; tokens em query string são recusados para não vazarem em histórico/logs. O proxy same-origin do Plano 4 injeta o header também para imagens e downloads;
- `GET /v1/jobs/{id}` com `result: {sha256, url}` quando `succeeded`, `checks` estáticos + medidos e erro PT-BR quando `failed`;
- a rota de assets de revisão cumpre o contrato de URLs do Plano 2: `fonts/{fileSha256}` (fonte por hash, sem extensão) e `sha256/ab/cd/{hash}` (blob de imagem de conteúdo) resolvem sob `assetsBaseUrl = /v1/brand-revisions/{id}/assets`;
- CLI `brand-api` (`serve`, `worker`, `create-token`).

**Tech Stack:** Python 3.12+ (dev em 3.14), FastAPI, SQLAlchemy 2 + psycopg 3, python-multipart, typer, pytest + TestClient (httpx), ruff. Postgres 16 local via Docker para dev/teste.

## Global Constraints

- Windows dev: paths só via `pathlib`; testes não podem depender de `/tmp`, usar `tmp_path`/`tmp_path_factory`.
- Sem rede em runtime e em testes, **exceto** o Postgres local (porta 5433 de teste). Fixtures binárias geradas programaticamente (PDF via PyMuPDF, PNG via Pillow, TTF via fontTools, ZIP via zipfile) — nenhum binário commitado.
- `packages/engine` é dependência read-only: instalar editable e **nunca modificar**; bug no motor = desvio a reportar, não consertar aqui.
- JSON de resposta sempre camelCase; artefatos do motor (draft, IR, kit, checks) serializados com `model_dump(by_alias=True)` — a API não renomeia nem re-modela campos do motor.
- Strings visíveis ao usuário (mensagens `detail` de erro, erros de job): PT-BR.
- Sem Alembic no M1: `Base.metadata.create_all` na criação do app (decisão registrada; migrações chegam quando houver segunda versão de schema).
- Ambiente próprio: venv em `apps/api/.venv` (Task 1). Antes de cada commit, a partir de `apps/api`: `.venv/Scripts/python -m pytest -q` verde (Postgres de teste no ar — README) e `.venv/Scripts/python -m ruff check .` limpo. A suíte do engine (`packages/engine`) também deve continuar verde.
- Commits na branch `m1-walking-skeleton`, mensagem `feat(api): <resumo>`.
- YAGNI agressivo: apenas o esboço v0 + as extensões listadas acima. Sem PUT/PATCH/DELETE, sem paginação, sem workspaces/multi-tenant (só as colunas mínimas), sem rate limit e sem CORS custom (web sempre usa proxy same-origin).

### Task 2: Storage content-addressed

**Files:**
- Create: `apps/api/src/brand_api/storage.py`
- Test: `apps/api/tests/test_storage.py`

**Interfaces:**
- Produces:

```python
class Storage:
    def __init__(self, root: Path) -> None      # cria root se não existir
    def put(self, data: bytes) -> str            # grava e retorna sha256 hex (64 chars, minúsculo)
    def has(self, sha256: str) -> bool
    def get(self, sha256: str) -> bytes          # KeyError se ausente
    def path_for(self, sha256: str) -> Path      # caminho do blob (existente ou não)
```

- Regras normativas:
  1. layout em disco: `<root>/sha256/<hash[0:2]>/<hash[2:4]>/<hash>` (contrato do plano-mestre);
  2. escrita atômica: gravar em arquivo temporário no mesmo volume e `os.replace` para o destino (Windows-safe);
  3. `put` é idempotente: blob já existente → retorna o hash sem regravar;
  4. hash calculado em streaming (`hashlib.sha256` alimentado em chunks) — nunca confiar em hash informado pelo cliente.

### Task 3: Unpacker seguro de ZIP

**Files:**
- Create: `apps/api/src/brand_api/unzip.py`
- Test: `apps/api/tests/test_unzip.py`

**Interfaces:**
- Produces:

```python
class UnzipError(Exception): ...                 # mensagem PT-BR

ALLOWED_EXTENSIONS = {".pdf", ".svg", ".png", ".jpg", ".jpeg", ".ttf", ".otf", ".json"}

@dataclass
class UnpackResult:
    manifest: dict[str, str]                     # path relativo (com "/") -> sha256 do conteúdo
    ignored: list[str]                           # entries puladas por extensão não permitida

def safe_unpack(zip_bytes: bytes, dest: Path, *,
                max_entries: int = 200,
                max_unpacked_bytes: int = 200 * 2**20) -> UnpackResult
```

- Regras normativas (spec §5.3 — uploads são conteúdo hostil):
  1. bytes que não são um ZIP válido → `UnzipError("O arquivo enviado não é um ZIP válido.")`;
  2. mais de `max_entries` entries (diretórios contam) → `UnzipError("O pacote contém arquivos demais.")` — verificar ANTES de extrair qualquer coisa;
  3. total descomprimido acumulado > `max_unpacked_bytes` → `UnzipError("O pacote descompactado excede o tamanho máximo permitido.")`; descompressão em chunks de 64 KiB com contagem — nunca confiar em `file_size` declarado;
  4. nome de entry inválido → `UnzipError("O pacote contém caminhos de arquivo inválidos.")`. Inválido = após normalizar `\` para `/`: caminho absoluto (começa com `/` ou casa `^[A-Za-z]:`), qualquer segmento `..`, ou vazio. Entries de diretório (nome termina em `/`) são puladas sem erro;
  5. entry que é symlink (`(zinfo.external_attr >> 16) & 0o170000 == 0o120000`) → mesmo erro da regra 4;
  6. extensão fora de `ALLOWED_EXTENSIONS` (case-insensitive) → não extrair; registrar o nome em `ignored`;
  7. extraídos vão para `dest/<path relativo>` (subdiretórios criados); `manifest` traz o sha256 de cada arquivo extraído, calculado durante a escrita;
  8. em caso de qualquer `UnzipError`, `dest` não deve ficar com extração parcial — extrair para diretório temporário irmão e mover no final, ou limpar `dest` no erro.

    with zipfile.ZipFile(buf, "w") as zf:

- Produces (rotas):
  - `POST /v1/documents/{id}/exports`, body `{"format": "png"|"pdf"}` → 202 `{"jobId": "job_…"}`;
  - `GET /v1/jobs/{id}` → 200 com EXATAMENTE as chaves `{"id", "status", "result", "checks", "error"}`; `result` é `null` até `succeeded`, depois `{"sha256": "...", "url": "/v1/assets/<sha256>"}`; `checks` contém o verdict estático enquanto aguarda e o verdict estático + medido depois do worker; `error` é `null` ou mensagem PT-BR.
- Produces (CLI): `brand-api worker [--poll-seconds 1.0] [--once]` — constrói tudo de `Settings.from_env()` e roda `run_worker`; `--once` processa no máximo um job e sai.
- Separação de processos: `create_app` nunca exige Playwright nem `BRANDRT_RENDER_DIST`; a API apenas valida/enfileira. Quando `settings.fake_exporter=True`, expõe `app.state.exporter = FakeExporter()` só para testes. `run_worker` escolhe `FakeExporter` nesse modo ou exige `render_dist` e cria `PlaywrightExporter` no modo real.

- Regras normativas — endpoint de export:
  1. documento inexistente → 404;
  2. `format` fora de `png|pdf` → 422 (validação Pydantic);
  3. `pdf` só é aceito quando o `profile` do layout é `doc-a4`; `png` vale para qualquer perfil; violação → 422 `"Exporte PDF apenas para documentos (A4)."`;
  4. o guard é REEXECUTADO na criação do job; qualquer check `blocked` → 409 com body `{"detail": "O documento tem pendências do guard — corrija antes de exportar.", "checks": [...]}` (nunca exportar conteúdo bloqueado — spec §5.8);
  5. sem pendências → INSERT em `jobs` (`kind="export"`, `params={"format": ...}`, `status="queued"`, `checks=<checks estáticos>`) → 202.
- Regras normativas — `run_next_job`:
  1. claim: `select(Job).where(Job.status == "queued").order_by(Job.created_at, Job.id).with_for_update(skip_locked=True).limit(1)`; nenhum job → retorna `False` sem efeitos;
  2. marca `running` + `started_at`, COMMITA (lock liberado antes do export longo — outro worker não trava);
  3. monta o workdir em `settings.work_dir / job_id` com `build_export_workdir` e exporta para `out.<fmt>` nesse mesmo diretório;
  4. `ExportOutcome.checks` substitui `job.checks` e `document.checks`; só então os bytes do arquivo exportado vão para `storage.put` → `status="succeeded"`, `result={"sha256": sha, "url": f"/v1/assets/{sha}"}`, `finished_at`;
  5. `ExportRejected` → `status="failed"`, `checks=exc.checks`, `result=null`, `error="O render encontrou pendências — corrija antes de exportar."`; nenhum blob é publicado;
  6. qualquer outra exceção → `status="failed"`, `error=f"Falha no export: {exc}"`, `finished_at`; retorna `True` em todos os desfechos processados;
  7. workdir removido ao final (best-effort: falha de remoção não muda o status do job).
- Regras normativas — `build_export_workdir` (materializa as DUAS raízes de asset numa só, cumprindo as URLs do Plano 2):
  1. cada `(path, sha)` do `manifest` da revisão → `dest/<path>` com os bytes do storage;
  2. cada `FontToken` do IR com `file_sha256` → cópia extra em `dest/fonts/<sha>` (sem extensão — `${assetsBaseUrl}/fonts/${fileSha256}`);
  3. cada image value do content (path já reescrito na Task 10) → blob em `dest/sha256/<2>/<2>/<sha>`;
  4. retorna `dest`.
